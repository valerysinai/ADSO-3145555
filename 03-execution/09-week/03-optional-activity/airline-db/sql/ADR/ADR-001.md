### ERROR-01 — Carpetas DDL y DML están intercambiadas

**Severidad:** 🔴 Crítico  
**Archivos:** `sql/DDL/` y `sql/DML/`

**Descripción:**  
Los contenidos de las carpetas están al revés. El archivo de **CREATE TABLE** (definición del esquema, que es DDL) está dentro de la carpeta `DML`, y los archivos de **INSERT INTO** (carga de datos, que es DML) están dentro de la carpeta `DDL`.

| Carpeta | Archivo | Contiene | Debería estar en |
|---|---|---|---|
| `sql/DML/` | `00_schema.sql` | `CREATE TABLE ...` | `sql/DDL/` |
| `sql/DDL/` | `01_geography.sql` | `INSERT INTO ...` | `sql/DML/` |
| `sql/DDL/` | `02_identity.sql` | `INSERT INTO ...` | `sql/DML/` |
| `sql/DDL/` | `03_security.sql` | `INSERT INTO ...` | `sql/DML/` |
| `sql/DDL/` | `04_airline_airport.sql` | `INSERT INTO ...` | `sql/DML/` |
| `sql/DDL/` | `05_aircraft.sql` | `INSERT INTO ...` | `sql/DML/` |

**Corrección:** Renombrar las carpetas o mover los archivos a la ubicación correcta.

### ERROR-02 — Columna `airport_type` no existe en la tabla `airport`

**Severidad:** 🔴 Crítico  
**Archivo:** `sql/DDL/04_airline_airport.sql`

**Descripción:**  
El script de inserción incluye la columna `airport_type`, pero dicha columna **no fue definida** en el `CREATE TABLE airport` de `00_schema.sql`. El INSERT fallará con error de columna inexistente.

**Schema definido:**
```sql
CREATE TABLE airport (
    airport_id   uuid PRIMARY KEY,
    address_id   uuid NOT NULL,
    airport_name varchar(150) NOT NULL,
    iata_code    varchar(3),
    icao_code    varchar(4),
    is_active    boolean NOT NULL DEFAULT true,
    ...
    -- ❌ NO existe airport_type
);
```

**DML problemático:**
```sql
INSERT INTO airport (airport_id, address_id, iata_code, icao_code, airport_name, airport_type)
-- ❌ 'airport_type' no existe en la tabla → ERROR en ejecución
```

**Corrección:** Añadir la columna al schema o eliminarla del INSERT.
```sql
-- Opción A: agregar al schema
ALTER TABLE airport ADD COLUMN airport_type varchar(30);

-- Opción B: quitar del INSERT
INSERT INTO airport (airport_id, address_id, iata_code, icao_code, airport_name) VALUES ...
```

### ERROR-05 — `fn_get_miles_balance` usa nombres de columna y valores de enum incorrectos

**Severidad:** 🔴 Crítico  
**Archivo:** `sql/05_functions_user/user_functions.sql`

**Descripción:**  
La función `fn_get_miles_balance` referencia la columna `miles_amount` y los tipos `'CREDIT'`/`'DEBIT'`, pero el schema define la columna como `miles_delta` y el CHECK constraint acepta únicamente `'EARN'`, `'REDEEM'` y `'ADJUST'`.

**Schema real de `miles_transaction`:**
```sql
CONSTRAINT ck_miles_transaction_type CHECK (transaction_type IN ('EARN', 'REDEEM', 'ADJUST')),
-- columna: miles_delta integer NOT NULL
```

**Código erróneo en la función:**
```sql
CASE
    WHEN transaction_type = 'CREDIT' THEN miles_amount   -- ❌ 'CREDIT' no existe, columna es miles_delta
    WHEN transaction_type = 'DEBIT'  THEN -miles_amount  -- ❌ 'DEBIT' no existe
    ELSE 0
END
```

**Corrección:**
```sql
-- La columna miles_delta ya es positiva para EARN y negativa para REDEEM
-- según como se inserte. Simplemente sumar:
SELECT COALESCE(SUM(miles_delta), 0)
INTO v_balance
FROM miles_transaction
WHERE loyalty_account_id = p_loyalty_account_id;

-- O si los valores REDEEM se guardan positivos y se necesita lógica:
CASE
    WHEN transaction_type = 'EARN'   THEN  miles_delta
    WHEN transaction_type = 'REDEEM' THEN -miles_delta
    WHEN transaction_type = 'ADJUST' THEN  miles_delta
    ELSE 0
END
```

### ERROR-06 — Procedimiento `sp_register_checkin` es incompleto (solo placeholder)

**Severidad:** 🔴 Crítico  
**Archivo:** `sql/04_stored_procedures/procedures.sql`

**Descripción:**  
El cuerpo del procedimiento usa `...` como valores, lo cual no es SQL válido. Además, el campo `checked_in_at` es `NOT NULL` en el schema y no está siendo provisto.

**Código actual:**
```sql
CREATE OR REPLACE PROCEDURE sp_register_checkin(
    p_ticket_segment_id uuid,
    p_status_code varchar
)
LANGUAGE plpgsql AS $$
BEGIN
    INSERT INTO check_in (ticket_segment_id, check_in_status_id, ...)
    VALUES (...);   -- ❌ Placeholder, no es SQL válido
END;
$$;
```

**Corrección sugerida:**
```sql
CREATE OR REPLACE PROCEDURE sp_register_checkin(
    p_ticket_segment_id  uuid,
    p_status_code        varchar,
    p_user_id            uuid DEFAULT NULL,
    p_boarding_group_id  uuid DEFAULT NULL
)
LANGUAGE plpgsql AS $$
DECLARE
    v_status_id uuid;
BEGIN
    SELECT check_in_status_id INTO v_status_id
    FROM check_in_status WHERE status_code = p_status_code;

    IF v_status_id IS NULL THEN
        RAISE EXCEPTION 'Estado de check-in no válido: %', p_status_code;
    END IF;

    INSERT INTO check_in (
        ticket_segment_id,
        check_in_status_id,
        boarding_group_id,
        checked_in_by_user_id,
        checked_in_at
    ) VALUES (
        p_ticket_segment_id,
        v_status_id,
        p_boarding_group_id,
        p_user_id,
        NOW()
    );
END;
$$;
```

### ERROR-07 — Vista `vw_reservation_summary` es incompleta e inválida

**Severidad:** 🔴 Crítico  
**Archivo:** `sql/07_views/views.sql`

**Descripción:**  
La vista usa `...` como SQL válido (no lo es) y además referencia `a.airport_name` sin hacer JOIN con la tabla `airport`. El script no puede ejecutarse.

**Código actual:**
```sql
CREATE OR REPLACE VIEW vw_reservation_summary AS
SELECT r.reservation_id, p.first_name, p.last_name,
       f.service_date, a.airport_name AS origin   -- ❌ 'a' no está definida
FROM reservation r
JOIN reservation_passenger rp ON r.reservation_id = rp.reservation_id
JOIN person p ON rp.person_id = p.person_id
...;   -- ❌ '...' no es SQL válido
```

**Corrección sugerida:**
```sql
CREATE OR REPLACE VIEW vw_reservation_summary AS
SELECT
    r.reservation_id,
    r.reservation_code,
    p.first_name,
    p.last_name,
    f.service_date,
    ao.airport_name AS origin_airport,
    ad.airport_name AS destination_airport,
    rs.status_name  AS reservation_status
FROM reservation r
JOIN reservation_status rs      ON r.reservation_status_id = rs.reservation_status_id
JOIN reservation_passenger rp   ON r.reservation_id = rp.reservation_id
JOIN person p                   ON rp.person_id = p.person_id
JOIN ticket t                   ON t.reservation_passenger_id = rp.reservation_passenger_id
JOIN ticket_segment ts          ON ts.ticket_id = t.ticket_id
JOIN flight_segment fs          ON ts.flight_segment_id = fs.flight_segment_id
JOIN flight f                   ON fs.flight_id = f.flight_id
JOIN airport ao                 ON fs.origin_airport_id = ao.airport_id
JOIN airport ad                 ON fs.destination_airport_id = ad.airport_id;
```

### ERROR-08 — Self-join erróneo en consulta de `user_role` (análisis 3FN)

**Severidad:** 🟡 Lógico  
**Archivo:** `sql/02_normalization/nf3_analysis.sql`

**Descripción:**  
La condición del JOIN de `user_account` usa el alias `ur` (de `user_role`) en lugar de `ua`, creando un self-join que siempre es verdadero y no filtra correctamente.

**Código erróneo:**
```sql
FROM user_role ur
JOIN user_account ua    ON ur.user_account_id = ur.user_account_id  -- ❌ debería ser ua.user_account_id
JOIN security_role sr   ON ur.security_role_id = sr.security_role_id
```

La condición `ur.user_account_id = ur.user_account_id` siempre evalúa a `true` (una columna siempre es igual a sí misma), por lo que `ua` nunca filtra y el CROSS JOIN con `user_account` devuelve un producto cartesiano.

**Corrección:**
```sql
FROM user_role ur
JOIN user_account ua    ON ur.user_account_id = ua.user_account_id  -- ✅ alias correcto
JOIN security_role sr   ON ur.security_role_id = sr.security_role_id
```

### ERROR-09 — FULL JOIN anulado por INNER JOIN posterior en `full_join.sql`

**Severidad:** 🟡 Lógico  
**Archivo:** `sql/03_queries/full_join.sql`

**Descripción:**  
El propósito del FULL JOIN entre `customer` y `loyalty_account` es mostrar también los clientes sin cuenta de lealtad y las cuentas sin cliente. Sin embargo, el INNER JOIN inmediato con `person` sobre `c.person_id` elimina todas las filas donde `c` es NULL (provenientes del lado derecho del FULL JOIN), haciendo que el FULL JOIN sea equivalente a un LEFT JOIN en la práctica.

**Código erróneo:**
```sql
SELECT p.first_name, la.loyalty_account_id
FROM customer c
FULL JOIN loyalty_account la ON c.customer_id = la.customer_id
JOIN person p ON c.person_id = p.person_id;   -- ❌ INNER JOIN elimina filas donde c es NULL
```

**Corrección (dos opciones):**
```sql
-- Opción A: usar LEFT JOIN para person si el objetivo es preservar lealtad sin cliente
SELECT p.first_name, la.loyalty_account_id
FROM customer c
FULL JOIN loyalty_account la ON c.customer_id = la.customer_id
LEFT JOIN person p ON c.person_id = p.person_id;   -- ✅

-- Opción B: si solo se quieren customers sin cuenta (LEFT JOIN es suficiente)
SELECT p.first_name, la.loyalty_account_id
FROM customer c
JOIN person p ON c.person_id = p.person_id
LEFT JOIN loyalty_account la ON c.customer_id = la.customer_id;
```

### ERROR-10 — Literal `'...'` inválido como UUID en `union.sql`

**Severidad:** 🟡 Lógico  
**Archivo:** `sql/03_queries/union.sql`

**Descripción:**  
Las condiciones del WHERE usan el literal `'...'` como si fuera un UUID válido. PostgreSQL intentará castearlo a `uuid` y fallará, o filtrará siempre vacío si se deja como `varchar`.

**Código erróneo:**
```sql
SELECT contact_value AS contacto, 'email' AS tipo
FROM person_contact WHERE contact_type_id = '...'   -- ❌ '...' no es un UUID válido
UNION
SELECT contact_value, 'phone' FROM person_contact WHERE contact_type_id = '...';  -- ❌
```

**Corrección:**
```sql
-- Usar los UUIDs reales insertados en la tabla contact_type:
SELECT contact_value AS contacto, 'email' AS tipo
FROM person_contact
WHERE contact_type_id = 'cc000003-0000-0000-0000-000000000001'  -- EMAIL
UNION
SELECT contact_value, 'phone'
FROM person_contact
WHERE contact_type_id = 'cc000003-0000-0000-0000-000000000002'; -- MOBILE

-- O mejor aún, usar un subquery para no hardcodear UUIDs:
SELECT pc.contact_value AS contacto, ct.type_code AS tipo
FROM person_contact pc
JOIN contact_type ct ON pc.contact_type_id = ct.contact_type_id
WHERE ct.type_code IN ('EMAIL', 'MOBILE');
```

## Decisión

Se documentan todos los errores encontrados para ser corregidos antes de la ejecución del proyecto. Los errores críticos (ERROR-01 al ERROR-07) deben resolverse en orden, ya que los errores de schema (02, 03, 04) deben corregirse antes de correr los INSERTs, y las carpetas (ERROR-01) deben reorganizarse antes de cualquier pipeline de despliegue automatizado.

---

## Consecuencias

- Los scripts **no pueden ejecutarse exitosamente** en su estado actual debido a los errores críticos.
- El análisis de 3FN y las queries de ejemplo contienen bugs lógicos que producirían resultados incorrectos si se ejecutaran sobre datos reales.
- El modelo de datos subyacente (schema conceptual y relacional) está **bien diseñado** y cumple 3FN; los errores están en la implementación SQL, no en el diseño.
